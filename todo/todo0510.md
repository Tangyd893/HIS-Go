## 一、当前结论

项目当前已经进入**后端工程基线稳定、运行闭环待打通**阶段。和 0509 相比，本次复核确认 `gofmt` / `go vet` / `go test` / `go build` 均可通过，说明后端可编译性风险已经下降；Docker Compose 配置仍可解析；集成测试框架在默认跳过模式下可运行。

但项目距离“可稳定联调交付”还差三类关键工作：

1. **Proto 契约仍未对齐业务入参**：5 个 gRPC 方法仍缺必要字段或仍复用 `common.IdRequest`，导致 gRPC server 只能使用占位值。
2. **Docker 全栈未完成真实启动验收**：当前只验证了 compose 配置解析，没有跑通容器构建、数据库初始化、服务健康检查和 API 集成测试。
3. **薄 service、错误码、MQ、ready 检查等横切能力仍偏原型化**：业务边界已有，但可靠性和运维可观测性还需要补强。

建议下一轮不要直接进入前端开发，先完成 **P0 运行闭环**：proto 补齐生成 + Docker 全栈 smoke test + 集成测试跑通。前端可在后端契约稳定后启动。

## 三、从 0509 延续下来的关键问题

### P0：Proto 契约与 gRPC 实现不一致

当前 0509 记录的 5 处 proto 缺口仍成立。部分返回结构已有字段，但请求结构仍缺字段，gRPC server 因此仍在使用占位值。

| 文件                    | 当前问题                                                                                     | 影响                                            |
| --------------------- | ---------------------------------------------------------------------------------------- | --------------------------------------------- |
| `registration.proto`  | `RegistrationRequest` 只有 `patient_id`、`schedule_id`，缺 `patient_name`、`registration_date` | gRPC 挂号无法传完整患者名与挂号日期                          |
| `pharmacy.proto`      | `DispenseRequest` 只有 `prescription_id`，缺 `drug_id`、`quantity`、`dispenser_id`             | gRPC 发药仍需占位值                                  |
| `examination.proto`   | `ReviewReportRequest` 缺 `reviewer_id`                                                    | 检查报告审核人无法从 gRPC 请求进入业务层                       |
| `health_record.proto` | `RevokeAuthorization` 使用 `common.IdRequest`                                              | service 需要 `patient_id + doctor_id`，当前无法完整表达  |
| `emr.proto`           | `QualityControl` 使用 `common.IdRequest`                                                   | 质控需要 `reviewer_id`、`level`、`comment`，当前无法完整表达 |

优化动作：

1. 补齐上述 request message，保留字段编号向后兼容，避免破坏已生成结构中的既有编号。
2. 安装并固定 proto 工具链，执行 `backend/scripts/proto_gen.sh` 重新生成 `.pb.go` / `_grpc.pb.go`。
3. 修改对应 `internal/*/grpc_server.go`，去掉 `"system"`、`1`、空字符串等占位值。
4. 增加 gRPC server 映射测试或至少 service 入参转换测试，防止后续 proto 再次漂移。

验收标准：

- `bash scripts/proto_gen.sh` 可执行成功。
- `rg "当前缺少|暂用固定值|待 proto" backend/internal` 无遗留占位注释。
- `go test ./... && go build ./cmd/...` 通过。

### P0：Docker 全栈启动与 API 验收未跑通

当前 Docker 仅验证了 compose 配置解析，尚未证明以下链路可用：

- 基础设施容器启动：PostgreSQL、Redis、RabbitMQ、Nacos、MinIO。
- 数据库初始化：`db_init.sh` 或 SQL 初始化脚本真实执行。
- 18 个 Go 服务镜像构建和容器启动。
- Gateway 到下游服务的路由、认证、JWT 透传。
- `testing/run.sh` 的 8 个 API 验收用例真实执行。

优化动作：

1. 用干净环境执行：
   
   ```bash
   cd docker
   cp .env.example .env
   docker compose up -d postgresql redis rabbitmq nacos minio
   cd ../backend
   bash scripts/db_init.sh
   cd ../docker
   docker compose up -d --build
   cd ../testing
   bash run.sh
   ```

2. 记录失败点，优先修复容器内环境变量、数据库连接、JWT PEM、服务发现地址、健康检查。

3. 增加一个 smoke test 文档或脚本，输出 `/health`、`/ready`、登录、受保护接口访问的最小结果。

验收标准：

- 所有核心容器状态为 healthy/running。
- Gateway `/health`、auth `/health`、user `/health` 可访问。
- `HIS_INTEGRATION_TEST=true go test -v ./...` 真实通过。

### P0：CI 与 Go 版本需要统一

`backend/go.mod` 当前声明 `go 1.25.0`，但 GitHub Actions 使用 `go-version: "1.24"`，README 也写 Go 1.24+。这会让 CI 结果依赖 Go toolchain 自动下载行为，稳定性不够。

优化动作：

1. 统一选择项目 Go 基线，建议设为 `1.25.x` 或当前可用的稳定版本。
2. 同步修改 `.github/workflows/ci.yml`、README 环境要求、开发脚本说明。
3. 如希望自动固定工具链，可在 `go.mod` 明确 `toolchain`，或在 CI 中设置对应 Go 版本。

验收标准：

- CI 配置、README、`go.mod` 三处版本口径一致。
- CI 不依赖隐式 toolchain 下载即可跑通。

## 四、P1 优化路线

### 4.1 补厚薄 service 层

以下模块当前仍大量直接透传 repository，测试也尚未覆盖，应作为下一轮业务质量建设重点：

| 模块                      | 当前状态                | 建议补强                             |
| ----------------------- | ------------------- | -------------------------------- |
| `inpatient/service`     | 8 个方法基本全透传          | 入院/出院状态机、押金校验、医嘱状态、护理记录时间校验      |
| `notification/service`  | 发送、批量发送、模板管理偏透传     | 模板参数渲染、接收人空值校验、发送状态、MQ/outbox 对接 |
| `health_record/service` | 授权和时间轴缺少业务约束        | 授权有效期、撤销幂等、医生/患者参数校验             |
| `statistics/service`    | 返回类型直接引用 repo 层 DTO | service DTO 解耦、日期范围校验、空结果语义      |
| `system/service`        | 字典/参数/日志管理偏透传       | 字典唯一性、参数 key 约束、操作日志审计字段补全       |

建议顺序：

1. `health_record`：直接关联授权安全，且 proto 也需要补。
2. `notification`：为 MQ 可靠性闭环做前置。
3. `inpatient`：业务状态机复杂，适合补单测。
4. `system` / `statistics`：补齐管理后台和报表基础能力。

### 4.2 统一 HTTP 错误语义

`pkg/response.Fail` 和部分错误处理当前业务失败仍返回 HTTP 200，只在 JSON body 中表达业务码。这样对网关、监控、前端拦截器和集成测试都不够友好。

优化动作：

- 增加错误码到 HTTP status 的映射，例如参数错误 400、未认证 401、无权限 403、资源不存在 404、业务冲突 409、内部错误 500。
- 保留统一 JSON body，但让 HTTP status 与错误类型一致。
- 更新集成测试，不再用“HTTP 200 且 code 非 0”兼容错误，而是明确断言 401/400 等状态。

### 4.3 统一 `/ready` 依赖检查

目前 `pkg/health` 已提供 readiness handler，但主要只有 gateway/auth 使用；多数服务仍只有简单 `/health`。README 中“`/ready` 就绪检查含 DB、Redis”描述需要和真实代码对齐。

优化动作：

- 为所有依赖 DB 的服务挂载 `/ready`。
- 对依赖 Redis、RabbitMQ 的服务补对应检查。
- 约定 `/health` 只表示进程存活，`/ready` 表示依赖可用。

## 五、P2 可靠性与可观测性

### 5.1 MQ 可靠发送闭环

`pkg/mq/rabbitmq.go` 已启用 Publisher Confirm，也提供 `NotifyConfirm()`，但 `Publish()` 只调用 `ch.Publish`，业务层没有等待 confirm；`billing` 里还存在 `_ = s.mq.Publish(...)` 忽略错误的情况。

优化动作：

- 封装 `PublishWithConfirm(ctx, ...)`，等待 ack/nack 或超时。
- 业务层禁止忽略 MQ 发布错误，必要时写入 outbox。
- 引入 `msg_record` 的写入、扫描、重试、终态标记。
- 为 billing 支付成功、退款成功、pharmacy 过期药品通知建立最小 outbox 流程。

### 5.2 可观测性接入

当前已有 tracing/recovery/ratelimit 等中间件雏形，但 OpenTelemetry、Prometheus、Grafana 仍未落地。

优化动作：

- 先补 Prometheus `/metrics`，覆盖 HTTP 请求量、延迟、错误率、gRPC 调用量。
- 再接入 OpenTelemetry trace id，并把 trace id 写入日志。
- Docker Compose 增加 prometheus/grafana profile，避免默认开发启动过重。
