## 一、项目概述

本项目是对原 [Hospital-Information-System](https://github.com/Tangyd893/Hospital-Information-System)（Spring Cloud Alibaba + Vue.js）的 **Go 语言重构版本**，全链路医院信息系统（Hospital Information System，HIS），覆盖**院内诊疗流程**与**院外患者服务**两大核心板块。系统采用前后端分离的微服务架构，后端全面迁移至 Go 生态，所有服务统一通过 Docker 进行容器化部署与管理，旨在构建一套高可用、高性能、易扩展的智慧医疗平台。

### 1.1 重构目标

- 将 17 个 Java 微服务模块完整迁移为 Go 微服务
- 保持原有业务逻辑与数据库设计不变，仅更换技术栈
- 利用 Go 协程模型提升并发处理能力，降低资源消耗
- 通过 gRPC 替代 OpenFeign，获得更强的类型安全与跨语言支持
- 保持消息可靠性设计（RabbitMQ），解决重复消费与消息丢失问题
- 高频数据（号源、排队状态）通过 Redis 缓存，保证系统响应速度

### 1.2 设计原则

| 原则       | 说明                                             |
| ---------- | ------------------------------------------------ |
| 服务自治   | 各微服务独立开发、独立部署、独立扩展             |
| 高可用     | 关键服务多副本部署，数据库主从架构               |
| 高性能     | Redis 缓存高频数据，RabbitMQ 异步解耦耗时操作    |
| 安全合规   | 符合医疗数据隐私保护相关法规要求                 |
| 可观测性   | 全链路日志追踪、指标监控、告警体系               |

### 2.2 调用链路

```
外部请求 → Gateway (路由/鉴权) → 业务微服务 → 数据层(PostgreSQL/Redis)
                                                  │
                                          RabbitMQ (异步解耦)
                                                  │
                                         下游消费者服务 (通知/统计/日志)
```

微服务间同步调用使用 **gRPC**（高性能二进制协议）和 **RESTful HTTP**（对外API），异步解耦使用 **RabbitMQ**。所有服务注册与发现由 **Nacos** 统一管理，配置中心同样基于 Nacos 实现动态配置热更新。

## 五、院内诊疗流程模块

### 5.1 核心业务流程

```
患者挂号 → 分诊叫号 → 医生接诊 → 检查/检验申请 → 检查执行/报告
                │                      │
                │              ┌───────┴────────┐
                │              │  处方开具/审核   │
                │              └───────┬────────┘
                │                      │
                └──────────────────────┼─────────── 收费结算 ─── 药房发药
                                       │                      │
                                       └──── 住院（入院/护理/出院）
```

### 5.2 关键功能模块详解

#### 5.2.1 挂号预约服务 (`his-registration`)

- **号源管理**：每日号源由排班服务定时生成，存入 PostgreSQL，同步至 Redis（Hash 结构）
- **预约挂号**：患者在线预约/现场挂号，扣减号源时使用 Redis 分布式锁防止超卖
- **排队叫号**：排队状态以 Redis Sorted Set 存储，按挂号时间排序，支持实时更新叫号进度
- **Redis 缓存策略**：
  - 号源缓存：`hash:schedule:{deptId}:{date}` → `{timeSlot: remainCount}`
  - 排队队列：`zset:queue:{deptId}:{date}` → `{{registrationId: timestamp}}`
  - 分布式锁：`lock:registration:{scheduleId}`，TTL 30s

#### 5.2.2 门诊诊疗服务 (`his-clinic`)

- 医生接诊登记、就诊时间记录
- 诊断录入（ICD-10 国际疾病编码）
- 检查/检验申请
- 历史就诊记录查询、过敏史预警提示
- 病历书写功能已剥离至 `his-emr`（详见《EMR系统补充计划书》）

#### 5.2.3 电子病历服务 (`his-emr`)

- SOAP 结构化病历书写（主诉、现病史、既往史、体格检查、辅助检查、诊断、处理计划）
- 病历模板引擎（专科/疾病个性化模板）
- 三级质控流程（时限/完整性/逻辑质控）
- CDSS 临床决策支持（药物相互作用、过敏史校验、剂量合理性）
- HL7 FHIR R4 标准化数据交换接口

#### 5.2.4 处方管理服务 (`his-prescription`)

- 西药/中成药/中草药处方开具
- 处方合理性审核（配伍禁忌、剂量校验、医保适应症校验）
- 处方状态流转：`草稿 → 待审核 → 已审核 → 已收费 → 已发药`
- 电子签名集成

#### 5.2.5 药房管理服务 (`his-pharmacy`)

- 药品入库/出库/盘点管理
- 发药窗口配药与核对
- 效期预警（cron 定时扫描即将过期药品并推送告警消息）
- 库存不足预警及自动补货建议

#### 5.2.6 收费结算服务 (`his-billing`)

- 挂号费、检查费、药费、治疗费多类型费用合并结算
- 医保实时结算（通过外部医保接口）
- 微信/支付宝扫码支付
- 退费流程（原路退回，需审批）

#### 5.2.7 住院管理服务 (`his-inpatient`)

- 入院登记 → 床位分配 → 医嘱执行 → 护理记录 → 出院结算
- 住院费用日清单
- 转科、转床处理

## 六、院外患者服务模块

### 6.1 核心服务

#### 6.1.1 在线问诊 (`his-outpatient`)

- 图文咨询（患者上传症状描述 + 图片，医生异步回复）
- 视频问诊（集成第三方音视频 SDK，如 TRTC）
- 问诊消息通过 RabbitMQ 异步推送，保证消息时序性
- 问诊结束后自动生成问诊记录，同步至健康档案

#### 6.1.2 慢病管理 (`his-outpatient`)

- 慢病患者签约建档（高血压、糖尿病等）
- 周期性健康数据上报（血压、血糖自测值）
- 异常数据自动预警推送给签约医生
- 用药提醒、复诊提醒定时推送

#### 6.1.3 健康档案服务 (`his-health-record`)

- 患者全生命周期健康档案汇总：就诊记录、检查报告、处方记录、随访记录
- 档案数据脱敏展示，支持患者授权查看
- 对接区域卫生信息平台（HL7/FHIR 标准）

#### 6.1.4 随访管理服务 (`his-followup`)

- 诊后随访计划自动生成（基于疾病类型推荐模板）
- 随访任务分派（电话随访/问卷随访/上门随访）
- 满意度调查问卷配置与收集
- 康复指导方案推送

## 七、数据层设计

### 7.1 PostgreSQL 数据库

- **数据库版本**：PostgreSQL 17
- **架构模式**：主从复制（1 主 + 1/2 从），读写分离
- **分库策略**：按业务域垂直拆分，每个微服务独立数据库（Database per Service）
- **Go 驱动**：`pgx` (高性能纯 Go PostgreSQL 驱动) + GORM

| 微服务           | 数据库名               |
| ---------------- | ---------------------- |
| his-user         | his_user               |
| his-auth         | his_auth               |

## 八、RabbitMQ 消息可靠性设计

### 8.1 消息队列使用场景

| 场景                 | 交换机类型 | 说明                                           |
| -------------------- | ---------- | ---------------------------------------------- |
| 挂号成功通知         | Topic      | 挂号成功后异步发送短信/推送通知，患者与医院双侧通知 |
| 处方状态变更通知     | Topic      | 处方审核通过/驳回后通知医生和患者               |
| 检查报告发布通知     | Direct     | 报告完成后通知患者查看                         |
| 排队叫号更新         | Fanout     | 诊室叫号状态变化同步到各显示终端（大屏/诊间屏） |
| 住院费用日清单       | Direct     | 每日定时生成并推送至患者端                     |
| 随访任务分发         | Direct     | 随访计划到期时创建并分配随访任务               |
| 慢病异常告警         | Topic      | 患者自测数据异常时告警通知签约医生             |
| 操作审计日志         | Fanout     | 所有敏感操作日志异步持久化                     |

### 8.2 消息可靠性三大保障策略

#### 8.2.1 确保消息正常发送（生产者可靠发送）

```
┌───────────────────────────────────────────────────────────┐
│                    消息发送流程                             │
│                                                            │
│  业务服务                                                │
│    │                                                      │
│    │ ① 业务数据入库（GORM 事务）                          │
│    ▼                                                      │
│  ┌───────────────┐   ② 写入消息本地记录表                  │
│  │  msg_record   │     (与业务数据同一事务)                │
│  │  本地消息表    │                                        │
│  └───────┬───────┘                                        │
│          │ ③ 发送消息到 RabbitMQ                          │
│          ▼                                                │
│  ┌───────────────┐                                        │
│  │   RabbitMQ    │  ← 启用 Publisher Confirm 机制         │
│  └───────┬───────┘                                        │
│          │ ④ 收到 Confirm 回调                             │
│          ▼                                                │
│  ┌───────────────┐   ⑤ 标记消息状态为"已发送"             │
│  │  msg_record   │                                        │
│  └───────────────┘                                        │
│                                                            │
│  ⑥ Cron 定时扫描"待发送"状态且超时的消息 → 重试发送      │
└───────────────────────────────────────────────────────────┘
```

**具体措施：**

1. **本地消息表（Transactional Outbox 模式）**：
   - 在业务数据库中以同一事务写入业务数据和消息记录表（`msg_record`），利用 GORM 事务保证数据一致性
   - 消息状态：`待发送(0)` → `已发送(1)` → `消费成功(2)` / `消费失败(3)`

2. **Publisher Confirm（发布确认）**：
   - 使用 `amqp091-go` 的 `ch.Confirm(false)` 启用发布确认模式
   - 通过 channel 异步接收 `amqp.Confirmation`，ACK/NACK 分别处理
   - Broker 返回 Confirm ACK 后，更新本地消息表状态为"已发送"
   - Broker 返回 Confirm NACK 后，触发消息重试

3. **消息落盘与补偿**：
   - RabbitMQ 队列和消息均设置为持久化（`durable=true`, `DeliveryMode: amqp.Persistent`）
   - 使用 `robfig/cron` 定时任务扫描本地消息表中超过 N 分钟仍为"待发送"状态的记录，重新投递

4. **消息发送失败补偿**：
   - 发送失败自动重试 3 次，间隔递增（2s / 5s / 10s），Go 协程控制
   - 超过最大重试次数后记录到死信表（`msg_dead_letter`），触发告警通知运维

#### 8.2.2 防止消息丢失（消息持久化 + 手动确认）

```
生产者 ──→ Exchange ──→ Queue ──→ 消费者
  │            │           │           │
  │   持久化    │   持久化   │   持久化   │  手动ACK
  │   Confirm   │           │           │
  ▼            ▼           ▼           ▼
保证消息       |           |           消费成功
发送成功    Exchange      Queue      before ACK
           Mirror模式   Mirror模式
```

**具体措施：**

1. **交换机与队列持久化**：
   - 所有交换机声明时设置 `durable=true`
   - 所有队列声明时设置 `durable=true`
   - 消息投递时设置 `amqp.Persistent`

2. **镜像队列（Quorum Queue）**：
   - 使用 RabbitMQ Quorum Queue 替代传统镜像队列
   - 基于 Raft 共识协议，消息在多数节点确认写入后才返回成功
   - 单节点故障不影响消息可用性

3. **消费者手动确认（Manual ACK）**：
   - 设置 `autoAck=false`，消费者**必须在业务逻辑处理成功后**手动调用 `msg.Ack(false)`
   - **严禁自动确认**，避免消费失败时消息丢失
   - `prefetch=1`，每次拉取一条，保证公平分发

4. **消费失败重试与死信队列（DLQ）**：
   - 消费失败后不直接否定确认（NACK），而是进入重试队列
   - 重试策略：重试 3 次，间隔 3s / 10s / 30s
   - 超过重试上限后，消息路由至死信队列（`queue.dlx`），由运维人工处理

```
重试流程：
消息消费失败
    │
    ▼
重试队列(retry) ─── 3次重试 ─── 仍失败 ──→ 死信队列(DLX)
    │                                          │
    │ 成功                                     ▼
    ▼                                    人工干预/告警
手动ACK确认
```

#### 8.2.3 防止重复消费（幂等性设计）

```
消费者收到消息
    │
    ▼
检查 msg_record 表中消息状态
    │
    ├── 状态=已消费 → 直接 ACK，跳过业务处理（已消费，幂等返回）
    │
    ├── 状态=处理中 → 重新入队或延迟重试（并发场景，Go sync.Mutex 防护）
    │
    └── 状态=待消费 → 执行业务逻辑
                         │
                    ┌────┴────┐
                    ▼         ▼
               业务成功     业务失败
                    │         │
                    ▼         ▼
         更新状态=已消费   更新状态=消费失败
            手动ACK        NACK 进入重试
```

**具体措施：**

1. **消息唯一标识**：
   - 每条消息携带全局唯一 ID（`msgId`），由生产者生成（Go 实现雪花算法）
   - 消费者在本地消息表中以 `msgId` 为唯一键

2. **数据库唯一约束**：
   - `msg_record` 表对 `msg_id` 字段建立唯一索引（`UNIQUE INDEX`）
   - 消费前先执行 `INSERT INTO msg_record(...) ON CONFLICT DO NOTHING`，利用唯一约束保证仅处理一次
   - 若插入返回 `RowsAffected == 0`，判定为重复消息，直接 ACK 跳过

3. **业务幂等性**：
   - 对于无法依赖数据库唯一约束的场景，在业务层面实现幂等：
     - 挂号扣减号源：使用 Redis `DECR` 原子操作 + 分布式锁
     - 处方状态更新：使用乐观锁（`version` 字段 + `WHERE version = ?`）
     - 收费状态变更：通过业务流水号（`billNo`）做幂等判断

4. **Redis 防重标记**：
   - 消费开始前，执行 `SETNX msg:consumed:{msgId} 1 EX 86400`
   - 若 `SETNX` 返回 `false`，说明消息已被消费，直接 ACK 跳过

5. **消息过期时间**：
   - 每条消息设置合理的 TTL（如 24 小时），超时自动转入死信队列
   - 避免历史消息堆积导致重复消费误判

## 十一、安全设计

### 11.1 认证鉴权流程

```
用户登录 → Gateway 转发 → Auth 服务验证
                              │
                              ├── 验证通过 → 生成 JWT Token（含用户ID、角色、权限列表）
                              │                → Token 存入 Redis（支持主动失效）
                              │
                              └── 验证失败 → 返回 401
                              │
后续请求 → Gateway Gin 中间件 → 解析 JWT → 校验签名与有效期
                              │                → 从 Redis 校验 Token 有效性
                              │                → 注入用户上下文到请求 Header
                              │
内部服务 → gRPC 拦截器 → 自动透传用户上下文（metadata 传递）
```

### 11.2 Go 安全实现要点

- JWT Token 使用 RS256 非对称加密（golang-jwt）
- 敏感接口（处方开具、收费等）实施二次认证
- 操作日志全程记录（Gin 中间件 + RabbitMQ 异步持久化）
- SQL 注入防护（GORM 参数化查询，避免字符串拼接）
- XSS 过滤（前端输入校验 + 后端 `html.EscapeString`）
- API 限流（基于 Redis 滑动窗口算法的限流中间件）
- 数据库密码、API 密钥等敏感配置通过环境变量注入，禁止硬编码

### 11.3 安全措施一览

- JWT Token 使用 RS256 非对称加密
- 敏感接口（处方开具、收费等）实施二次认证
- 操作日志全程记录（Gin 中间件 + RabbitMQ 异步持久化）
