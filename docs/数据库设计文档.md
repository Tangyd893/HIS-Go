# HIS-Go 数据库设计文档

## 一、数据库架构

采用 **Database per Service** 分库策略，每个微服务拥有独立的 PostgreSQL 17 数据库：

| 数据库 | 对应服务 | 主要表 |
|--------|---------|--------|
| `his_auth` | 认证服务 | users, roles, permissions, role_permissions, user_roles, login_logs |
| `his_user` | 用户服务 | patients, employees, departments |
| `his_registration` | 挂号服务 | schedules, registrations, queue_records |
| `his_clinic` | 门诊服务 | clinic_records, diagnoses, examination_requests |
| `his_emr` | 电子病历 | medical_records, record_sections, quality_controls, record_templates |
| `his_prescription` | 处方服务 | prescriptions, prescription_details, prescription_reviews |
| `his_billing` | 收费服务 | bills, bill_details, payments, refunds |
| `his_pharmacy` | 药房服务 | drugs, drug_stocks, stock_records, dispense_records |
| `his_examination` | 检查服务 | examination_reports, report_reviews |
| `his_inpatient` | 住院服务 | inpatient_records, beds, medical_orders, nursing_records |
| `his_schedule` | 排班服务 | doctor_schedules, room_schedules |
| `his_outpatient` | 院外服务 | consultations, messages, contracts, health_data |
| `his_followup` | 随访服务 | followup_plans, followup_tasks, surveys |
| `his_health_record` | 健康档案 | health_records, timeline_events, authorizations |
| `his_notification` | 通知服务 | notification_templates, notification_records |
| `his_statistics` | 统计服务 | operation_stats, revenue_records, dept_workloads |
| `his_system` | 系统服务 | dict_types, dict_items, system_params, operation_logs |

## 二、核心表结构

### 2.1 patients (his_user)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| name | VARCHAR(100) | 患者姓名 |
| gender | CHAR(1) | M-男 F-女 |
| birth_date | DATE | 出生日期 |
| phone | VARCHAR(20) | 手机号 |
| id_card | VARCHAR(18) | 身份证号（唯一） |
| address | TEXT | 住址 |
| allergy_history | TEXT | 过敏史 |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |
| deleted_at | TIMESTAMP | 逻辑删除 |

### 2.2 schedules (his_registration)

| 字段 | 类型 | 说明 |
|------|------|------|

|------|------|------|

| 字段 | 类型 | 说明 |

## 三、设计规范

1. **主键**：统一使用 UUID 类型，`DEFAULT gen_random_uuid()`
2. **乐观锁**：涉及并发修改的表增加 `version` 字段
3. **软删除**：需要逻辑删除的表增加 `deleted_at TIMESTAMP` 字段
4. **审计字段**：所有表包含 `created_at`、`updated_at`
5. **索引**：外键字段、查询条件字段建立索引
6. **唯一约束**：身份证号、账单号、处方号等业务唯一字段
