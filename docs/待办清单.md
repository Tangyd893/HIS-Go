## 〇、项目目标（当前阶段）

本阶段**不以生产级全功能 HIS 上线**为目标，而是聚焦 **两个前端各自可独立演示**，后端 **可部署到云服务器** 为上述演示提供 API，并遵循 **尽量轻量化** 原则：

| 目标 | 前端 | 演示形态 | 后端要求 |
|------|------|----------|----------|
| **目标 A** | Web 医院信息管理系统 | 浏览器访问 `his-web-admin-react`（`/admin`） | 云服务器部署 **管理端演示 Profile**，只启动管理演示所需微服务 |
| **目标 B** | 小程序患者端 | 微信开发者工具 / 真机 → `his-mp-webview` → web-view 加载 `his-web-patient-react` | 云服务器部署 **患者端演示 Profile**，只启动患者演示所需微服务 |

### 核心原则

1. **分场景启动**：演示管理端时不启动患者专用服务；演示小程序时不启动住院/EMR/统计等管理专用服务（除非该页面纳入患者演示脚本）。
2. **共享最小基础设施**：PostgreSQL + Redis + Gateway 为两套 Profile 共用；RabbitMQ 仅在管理端 Profile 需要（billing/pharmacy）；Nacos / MinIO **演示阶段默认不启**。
3. **演示 ≠ 全量**：每个 Profile 先打通 **一条可讲解的业务主路径**，再按需扩展「页面可浏览」。
4. **云部署优先**：本地开发与云端使用同一套 Profile 定义，避免「本地全启、云上装不下」。

## 二、演示架构设计

### 2.1 整体关系

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         云服务器（单节点轻量化）                            │
│  ┌──────────┐   ┌─────────────┐   ┌──────────────────────────────────┐  │
│  │ PostgreSQL│   │    Redis    │   │  Gateway :8080                   │  │
│  └──────────┘   └─────────────┘   │  + Profile 选启的微服务（见 §2.3） │  │
│  ┌──────────┐   （管理端 Profile   └──────────────────────────────────┘  │
│  │ RabbitMQ │    额外需要）         ┌──────────────────────────────────┐  │
│  └──────────┘                      │  Nginx :80/443                   │  │
│                                    │  /admin  → 管理端静态资源         │  │
│                                    │  /patient→ 患者 H5 静态资源       │  │
│                                    │  /api/   → Gateway               │  │
│                                    └──────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────────┘
         ▲                                        ▲
         │ HTTPS                                  │ web-view
         │                                        │
  ┌──────┴──────┐                          ┌──────┴──────┐
  │ 浏览器       │                          │ 微信小程序    │
  │ 管理端演示   │                          │ his-mp-webview│
  └─────────────┘                          └─────────────┘
```

### 2.2 基础设施分层

| 组件 | 管理端 Profile | 患者端 Profile | 说明 |
|------|:-------------:|:-------------:|------|
| PostgreSQL | ✅ | ✅ | 各微服务独立库，迁移脚本已就绪 |
| Redis | ✅ | ✅ | 各服务启动时硬依赖，不可省略 |
| Gateway | ✅ | ✅ | 统一 `/api` 入口 |
| RabbitMQ | ✅ | ❌ | 仅 billing、pharmacy 启动时需要 |
| Nacos | ❌ | ❌ | Gateway 静态路由即可，演示不启 |
| MinIO | ❌ | ❌ | 业务层未接入，演示不启 |

### 2.3 微服务 Profile 定义

> 计数不含 Gateway。全量生产为 17 个业务服务；演示 Profile 为子集。

#### Profile A：`demo-admin` — Web 管理端演示

**业务主路径（必须打通）**：登录 → 排班/挂号 → 门诊接诊 → 开具处方 → 收费 → 药房发药

| 微服务 | 端口 | 主路径 | 扩展页面 |
|--------|------|--------|----------|
| his-auth | 8081 | 登录 / Token | — |
| his-user | 8082 | 患者/员工/科室查询 | 用户管理 |
| his-schedule | 8090 | 排班列表 | 排班管理 |
| his-registration | 8083 | 号源、挂号、叫号 | 挂号管理 |
| his-clinic | 8084 | 接诊、诊断 | 门诊诊疗 |
| his-prescription | 8085 | 开方、审核 | 处方管理 |
| his-billing | 8086 | 结算、支付（Stub） | 收费管理 |
| his-pharmacy | 8087 | 库存、发药 | 药房管理 |
| his-system | 8096 | 字典、参数 | 系统管理 |

**A 档（核心演示，9 个业务服务 + Gateway）**：上表全部。  
**B 档（全页浏览，+8 个服务）**：+ examination、inpatient、emr、statistics、outpatient、followup、health_record、notification。

**基础设施**：PostgreSQL + Redis + **RabbitMQ** + Gateway + Nginx（托管 `/admin` 静态资源）。

**推荐云规格（A 档）**：2 vCPU / 4 GB RAM / 40 GB SSD（紧凑可跑；演示前需确认内存占用）。

**B 档（核心演示，8 个业务服务 + Gateway）**：auth、user、registration、schedule、prescription、examination、followup、health_record。

**不需要启动（患者 Profile）**：clinic、billing、pharmacy、inpatient、emr、statistics、system、notification。

**基础设施**：PostgreSQL + Redis + Gateway + Nginx（托管 `/patient` 静态资源）。**无需 RabbitMQ**。

**推荐云规格（B 档）**：2 vCPU / 4 GB RAM / 40 GB SSD。

## 六、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 全量 Compose 默认启动 | 2C4G 云机 OOM | 强制 Profile 化；README 演示章节指向 demo compose |
| 管理端 API 路径不一致 | 演示点击报 404 | A-F1 优先；临时可用 Vue 备份版演示 |
| 患者页 mock 数据 | 小程序「能开不能用」 | B-F2–B-F5 优先于 Dashboard 美化 |
| billing/pharmacy 强依赖 MQ | 管理 Profile 启动失败 | demo-admin compose 显式包含 rabbitmq |
| 小程序 web-view 要 HTTPS | 真机无法加载 HTTP | 云部署必须 C-3 + 业务域名；本地仅用开发者工具 |
| 双 Profile 共用同一 DB | 演示数据互相污染 | seed 脚本区分场景或演示前 `db reset` 文档化 |
| `/admin` `/patient` SPA 刷新 404 | 云上演示路由失败 | A-F3、B-F1 配置 base + basename |
